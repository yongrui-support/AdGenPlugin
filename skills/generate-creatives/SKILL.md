# generate-creatives

把品牌資訊變成可投放的廣告素材。核心產物是
**`composition_prompt`：一段強結構化的自然語言，貼進 GPT（或其他生圖工具）就能生出廣告主視覺**。
依使用者狀況分兩種模式：
- **已有文案，只補構圖**
- **連文案一起發想多組**

> 重要前提（實測得到）：新版 GPT 生圖模型的**繁體中文渲染已很強**，文字直接寫進 prompt 即可，
> 幾乎不糊不錯字。

## 流程

### 1. 開場：先問「要做到哪一步」（決定模式）
**第一件事就主動問使用者**（這是分流關鍵，別跳過）：

> 「你已經有廣告文案了嗎？還是文案也要一起發想？」

- **模式 A — 已有文案，只補構圖**：使用者提供現成文案（標題／內文／CTA），
  你**只為它產 `composition_prompt`**（圖中文字＝照搬使用者的文案，不亂改）。可一則文案產多個視覺版本。
- **模式 B — 連文案一起發想**：你產 **N 組（預設 5）「文案 ＋ 構圖 prompt」**，衝量提高可用率。

### 2. 收 brief（用「一步一步詢問」收集，不要一次丟一張表單）
逐步問、每次只問一小組，並附上預設值讓使用者能直接說「用預設」或補充；答完一步再問下一步。建議順序：

1. **品牌 / 產品**：`brand_name`、`product` —— 必要，先問這個。
2. **對誰說（受眾洞察——發想的真正起點，排在賣點之前）**：先搞清楚對象，鉤子才有對象、不是對空氣喊。**逐項都問**（使用者有就提供，沒有或不知道也沒關係，照常往下）：
   - **目標客群輪廓**（誰：年齡／身分／使用情境）。
   - **這位客群最痛的 1–2 件事**、**最想要的結果**。
   - **最常見的不買理由（objection）**：太貴／怕沒效／不適合我／不信任…（§3B 產 N 組廣告時，會**指定其中 1–2 組專門針對這些不買理由來寫**、直接化解抗拒）。
   - **真實顧客語言（VoC）**：「有沒有顧客評論／客訴／常見問題可以貼給我？」——有的話**直接把顧客的原話用進 `headline`／`primary_text`，越像顧客本人說的越好**（提煉、別又改回廠商腔）。
   - 這些**只用來引導發想**：照「額外要說的話」同規則，**不寫進 `brief`、不存進 JSON**（保持 schema 不變）。
3. **賣點 / 優惠**：`usp[]`(賣點)、`default_offer`(預設優惠，各組可在 content 蓋過)、`price`(可選)。
4. **調性 / 語言**：`brand_tone`(語氣)、`target_language`(預設 zh-TW)。
5. **合規 `compliance`**：該產業若有限制（禁用詞／必要揭露／年齡標示等）就問，沒有就留空。
6. **尺寸 `default_aspect`**：預設 `1:1`；可 `1:1` / `4:5` / `9:16` / `16:9`（看板生圖時各組可逐次換比例）。
7. **參考素材（務必問，別跳過）**：
   - **先幫使用者把品牌素材夾建好**：`mkdir -p "data/materials/<brand_name>"`（`-p` 冪等、**已存在不會報錯**；資料夾名 **=== `brand_name`、大小寫逐字**，看板靠這個過濾顯示）。**若同品牌之前做過、夾裡已有素材／`index.json` → 沿用既有的**，只回看／改名／補描述「這次新丟的」，別重建、別洗掉舊素材。
   - **請使用者把素材「直接丟進那個資料夾」**（產品照／品牌風格／競品參考圖／logo…）——**別再往對話框貼圖**（檔一多很累）。
   - **丟完後你務必逐張「回看圖」並改成有意義的檔名**：很多人的檔名是 `IMG_9302934.jpg` 這種亂編號，構圖引用／看板都看不懂——你**讀圖判斷內容**，`mv` 成看得懂的名（如 `切達乳酪蛋蛋餅實拍.jpg`）。素材名稱＝相對路徑（含品牌夾，如 `杜倫先生/切達乳酪蛋蛋餅實拍`）。
   - **看著圖寫一句外觀描述**，合併寫入 `data/materials/index.json`（`{名稱: 描述}`；無則建立、有則合併）——構圖 prompt 之後靠這句精準引用。看板會自動掃到資料夾裡的圖。
   - 若使用者**自己先建了**品牌夾但大小寫不符（`dutek`／`DUTEK`）就改名對齊 `brand_name`：Windows 只差大小寫要過渡暫名（`dutek` → `dutek_tmp` → `Dutek`）；改名後同步置換 `index.json` 的鍵，並 grep `data/creatives/*.json` 把既有 `creatives.materials` 的舊前綴換新。
   - **品牌設計準則 `design.md`（這個選項務必主動告訴使用者，每次都要講）**：使用者可在該品牌素材夾放一個 `data/materials/<品牌>/design.md`（**資料夾名同樣要 === `brand_name`、大小寫逐字**，否則生圖時找不到），寫該品牌的設計準則。**主動講明「你可以放一個 design.md 給 Claude 遵循」**；有的話，產 `composition_prompt` 時**必須先讀它、整段直接遵循**（見 §3C 視覺最高優先）；沒有也沒關係、照常進行。
     - **使用者不會寫也沒關係**：主動提議「要不要我依品牌／現有素材幫你草擬一份？」，骨架五段——**配色**（hex）／**字體**／**版面偏好**／**風格調性**／**必避免的元素**。（design.md 只寫視覺準則，別寫合規——合規走 `compliance` 欄位。）
8. **Logo（在「產 composition_prompt 之前」確認；有就放、沒有別硬塞）**：問品牌有沒有 logo、要不要放進廣告圖。
   - **沒有 logo、或使用者不想放 → 完全沒關係，構圖就不放 logo，別硬塞**（不是每張廣告都需要 logo）。
   - **有、且要放**的話：
     - 收進素材庫（同上一步，放 `data/materials/<品牌>/`，例 `Dutek/Logo`、`Dutek/Logo_白色版`）。
     - **問深色背景要用的「白色／反白版」logo 有沒有**——深底用白版、淺底用原版是生圖鐵則；沒有就提醒使用者：深色構圖建議另備白版，否則 logo 容易對比不清或被洗成單色。
     - 確認想擺的位置（左下／右下／置中…）、是否每組都要。
     - 敲定後，§3C 會把「logo 版本＋位置」寫進**有放 logo 的那幾組** `composition_prompt`（直接畫進、不留淨空區）。
9. **有沒有額外要跟 Claude 說的話（務必主動問、別跳過）**：任何想交代的產出指示（風格偏好、必避免的元素、特定要求…），沒有就略過。這是**給 Claude 用來引導產出的指示，不寫進 `brief`、也不存進輸出 JSON**。

收齊後簡述一次 brief 給使用者確認，再往下產。

依模式再各補問（**「要產幾組」一定要主動問使用者，別自己默默套預設**——這點很常被跳過）：
- 若使用者選擇**模式 A**：請使用者**貼現成文案**（每則的 `headline` / `primary_text` / `cta`），並**主動問「一則文案要幾個視覺版本？預設 2——用預設還是自訂？」**（`K`）。
- 若使用者選擇**模式 B**：**主動問「要產幾組？預設 5——用預設還是自訂？」**（`N`），再問切角來源（二選一）：
  - **貼參考**：使用者貼競品廣告文案／連結／截圖 → 當場拆解切角與鉤子。
  - **純品牌 brief**：沒有參考時，依「品類 + brief」自行提 2～3 個切角給使用者挑。

確認後再產。

### 3. 產出通則（安全 / 合規 / 結構，A、B 都適用）

**安全 / 合規**
- 可使用客戶提供的素材（logo、產品照、過往視覺等）。
- 不要捏造 brief 沒給的價格／折扣／保證（模式 A 也不得改寫使用者文案的承諾）。
- 遵守 `compliance`。

**產出結構**（先知道目標形狀，兩模式都填這個）
```json
{
  "id": "...",
  "created_at": "<UTC ISO>",
  "schema_version": 4,
  "mode": "copy_only | full",
  "brief": {
    "brand_name": "...",
    "product": "...",
    "default_offer": "...",
    "price": null,
    "target_language": "zh-TW",
    "brand_tone": "...",
    "usp": ["..."],
    "compliance": "...",
    "default_aspect": "1:1"
  },
  "creatives": [
    {
      "angle": null,
      "hook_formula": null,
      "funnel_stage": null,
      "copy": {
        "primary_text": "...",
        "headline": "...",
        "cta": "..."
      },
      "content": {
        "title": "...",
        "offer_badge": "...",
        "cta": "..."
      },
      "materials": ["素材名稱…（=檔名，看板生圖時會把這些素材圖附給模型；無則空陣列）"],
      "ai_platform": "chatgpt | gemini（生圖平台，預設 chatgpt；gemini 僅使用者在看板手動選）",
      "model": "5.5（chatgpt 預設最新；gemini 預設 3.1 Pro）",
      "thinking_effort": "高（chatgpt 預設最高；gemini 預設 延長）",
      "composition_prompt": "純視覺：主題/主體/場景/配色/光線/氛圍/比例＋每段文字的位置；文字一律用 {{content.欄位}} 佔位引用＋合規負面表列"
    }
  ]
}
```
- **`content` 與 `composition_prompt` 一定要切開**：
  - `content` ＝「圖中要出現的文字」（標題、優惠標、CTA…，依需要增減欄位），是可單獨調整的文字槽。
  - `composition_prompt` ＝**只描述視覺與文字位置**；文字一律寫成 `{{content.欄位}}`（如 `{{content.title}}`、`{{content.cta}}`），**不要把字串寫死在 prompt 裡**。
  - 好處：之後使用者改 `content` 的字，圖上的字就跟著換，視覺描述完全不用動。
- **各欄位用途**：
  - `copy` ＝ 投放後台的貼文文案。
  - `content` ＝ 畫在圖上的文字（與 `copy` 可不同）。
  - `materials` ＝ 參考素材名稱清單（=檔名；生圖時附給模型，無則空陣列）。
  - `ai_platform` / `model` / `thinking_effort` ＝ 這組生圖用哪個平台／模型／思考程度（見下方「生圖平台」）。
- **兩模式怎麼填**：
  - **模式 A**：`mode="copy_only"`，`copy`/`content` 取自使用者提供的文案，`angle/hook_formula/funnel_stage` 可為 `null`；同一則文案的多個視覺版本各算一筆 `creatives`。
  - **模式 B**：`mode="full"`，各筆填齊 `angle/hook_formula/funnel_stage`。
- **生圖平台／模型／思考（每組三欄，產出一律帶預設最高的一組）**：
  - `ai_platform`：`"chatgpt"`（預設）或 `"gemini"`。**產出一律帶 `"chatgpt"`**（品質最穩）；`"gemini"` 實測不穩、**僅保留給使用者在看板手動選**。
  - `model`：該平台的模型，**產出帶「該平台清單的第一個＝最新」**（別在這裡複寫具體版號，免得改版過時——權威清單在 `migrations.py` 的 `PLATFORM_OPTIONS`，看板下拉同此）。
  - `thinking_effort`：思考程度，**產出帶「該平台清單的第一個＝最高」**（同上清單）。
  - 使用者要換平台/模型/思考，在看板下拉自己調（generate-images 生圖時會切到對的平台＋模型＋思考程度，見該 skill §2）。

**範例**（填好的樣子；改 `content` 的字，圖上的字就跟著換，視覺描述不用動）

### 3A. 模式 A 產出（只補構圖）
為使用者的**每一則文案**產 `K` 個 `composition_prompt`，每個走**不同視覺方向**（場景／構圖／配色／氛圍要有差異，不是微調）。
- `content` 圖中文字**照搬使用者文案**（標題／CTA／優惠標…），不得擅自改寫或新增承諾；`composition_prompt` 用 `{{content.欄位}}` 引用它們、只描述視覺與位置。
跳過 §3B 的文案發想（模式 A 不需要）；`composition_prompt` 仍依 §3C 規格，再到 §4。

### 3B. 模式 B 產出 N 組（衝量 ＋ 真多樣性）

**先收斂、再衝量**（別一上來就鋪 N 組各說各話）：依 §2 的受眾與賣點，先給使用者 **1–2 個最強「核心命題」（single-minded：一句話主張＋一句理由）讓使用者拍板**，再圍著它展開多樣性執行。

內建這套「結構骨架」當**推理依據**（不是查表，是用它逼出差異）：
- **hook 公式**：痛點放大 / 模式中斷 / 好奇缺口 / 社會證明 / 急迫稀缺 / 利益直球 / 故事代入
- **漏斗階段**：TOFU（認知）/ MOFU（考慮）/ BOFU（轉換）
- **認知階段（潛在客對問題／解法知道多少——決定 lead 怎麼開頭）**：不知道有問題 → 知道問題 → 知道有解法 → 知道這項產品 → 只差臨門一腳。同一漏斗階段、認知不同，開頭就不同（不知道問題的先戳痛；已在比價的直接給差異化＋證據）。**這是每組「怎麼開頭」的鏡頭，逐組標明即可，不當排列維度。**
- **情緒觸發**：FOMO、貪好康、歸屬感、爽快感、懷舊、勝負慾…

**多樣性鐵則**：N 組之間，各取**不同的（切角 × hook 公式 × 漏斗階段）組合**——主力在「切角／利益 × hook」拉開差異（N 預設 5，別硬湊四維正交、會生出彆扭組合）；**禁止只是換句話說**，要用不同訴求覆蓋面。**其中至少 1–2 組要直接回應 §2 收到的「不買理由」。**
**視覺也要多樣**：N 組的 `composition_prompt` 視覺方向也要有別（場景／構圖／配色別都同一套，否則文案不同、版面雷同，測試價值打折）。

**文案撰寫鐵則（每組都要）**：
- **利益翻譯**：把賣點翻成「對客群（產品的使用者）的利益與意義」，不是規格陳述（feature → benefit → 對客群的意義）。
- **可信度支點**：每組至少一個具體支點（數字／社會證明／機制／保證），別只靠空泛形容詞。
- **鉤子前置**：`copy.primary_text` **第一行就要勾人**——Meta 約 125 字後會被「⋯顯示更多」截斷，鉤子別藏在後面。
- **CTA 對齊漏斗**：TOFU 軟（「了解更多」）／BOFU 硬（「立即購買」），對齊該組 `funnel_stage`。

每組依 §3 結構填齊欄位，並標明 `angle / hook_formula / funnel_stage`（逼出差異、方便人工挑）。**追溯性（不動 schema 的輕量解）**：把「認知階段／回應的不買理由」折進 `angle` 字串，例：`angle: "利益直球｜認知:已比價｜回應:太貴"`——讓產物與看板上看得出「這支在測什麼、給誰」（投手做 A/B 最需要）。`composition_prompt` 規格見 §3C。

### 3C. 共用：`composition_prompt` 規格

**視覺最高優先——品牌 `design.md`（若有）**：該品牌素材夾若有 `data/materials/<品牌>/design.md`，**先讀它，整個 `composition_prompt` 的配色／字體／版面／風格／避免項都直接遵循它**；與下面**視覺通則**（主題/配色/構圖等）衝突時以 `design.md` 為準。
> ⚠️ **但 design.md 只管「視覺設計」，不得覆蓋合規**：第 4 條的 `compliance`（負面表列／必要揭露…）是法律性**一票否決、永遠最高**（見 §4.3），design.md 不能援引來拿掉必要揭露或塞進禁用元素；logo 配色正確性同樣不被 design.md 蓋過。

不論模式，每個 `composition_prompt` 必須具體寫出：
1. 主題 / 主體 / 場景
2. 配色 / 光線 / 氛圍 / 比例（用 `default_aspect`）
3. **圖中文字（佔位引用）**：標明每段文字的**位置**，文字本身一律寫成 `{{content.欄位}}`（如 `{{content.title}}`、`{{content.cta}}`），實際字串放進該創意的 `content`；**不要把字串寫死在 prompt 裡**。
4. **合規負面表列**：依 `compliance` 明確寫「畫面不得出現…」（若有需求）。
5. **構圖邏輯（寫完逐段自檢，避免「人沒看向電視」這類矛盾圖）**：
   - **一個主行為**：一張圖只給一個搶焦點的行為——「看球＋打牌＋談笑」這種要挑一個當主行為，其餘降為背景氛圍。
   - **視線與朝向**：寫了人物「正在看 X」，就要同時寫明「視線／身體朝向 X」（必要時指定側面或背面入鏡）；若重點其實是人物互動，就**不要寫「在看 X」**，把 X 改成背景光源／氛圍。
   - **空間關係**：物件相對位置要成立（例：螢幕在人物前方，就不可能同時正面朝向觀眾）。
6. **參考素材（若有）**：該組要以某素材為視覺基準時，把**素材名稱（=檔名）**填進該組 `materials`，並在 prompt 裡
   **用素材的外觀描述具體指涉**（例：「產品外觀以附件素材『平衡車實照』為準——雙輪直立、白色機身」）。
   看板生圖會把勾選的素材圖一併附給模型；「複製」流則會提醒使用者手動附圖。
7. **Logo（§2 step 8 已先確認；品牌沒提供 logo 或使用者不想放 → 這條整個跳過、構圖不放 logo，別硬塞）**。**有要放**才做下面：
   - 把 logo **直接畫進構圖**（指定位置，如「左下角」），**不要寫「淨空區供 logo 後製」**。
   - 版本依背景明暗：**深色背景→白色／反白版（保留品牌輔色）、淺色背景→原版**；忠實配色、不變形、不留佔位框。
   - 把該 logo 素材名稱一併填進此組 `materials`。（與 generate-images §3 的 logo 雷區一致——避免兩個 skill 對不上。）

### 4. 品質驗證（開 subagent 當獨立審稿人）
產完後**用 Agent tool 開一個 subagent** 獨立驗稿——自己驗自己有盲點，獨立 context 的審稿人較挑剔。
**一個 subagent 驗整批**（不是每組開一個），把 brief、全部 creatives 與下列**驗收標準**原文給它；**該品牌若有 `data/materials/<品牌>/design.md`，也一併把它的內容給 subagent 當「視覺驗收標準」**。

**驗收標準**（要求逐組逐項回報「通過／不通過＋一句理由」）：
1. **鉤子**：把品牌名遮住後，title/headline 仍勾人（好奇／情緒／利益擇一成立）？只是陳述產品＝不過。
2. **多樣性**：把全批（切角 × hook × 漏斗）列表比對——組合重複＝後者汰換；兩組訴求相同只是換措辭＝汰換其一。
3. **合規（一票否決）**：逐組對照 `brief.compliance` 檢查 copy 與 composition_prompt；負面表列缺漏、
   或出現 brief 沒給的價格／折扣／保證／規格＝不過，**修到過才可交付**。
4. **圖面可行性**：content 槽位 ≤3、單槽過長（約 >12 字）需有理由；composition_prompt 必含
   文字位置、`{{content.x}}` 佔位、比例（該組有 logo 則含 logo 版本/位置、且不留淨空區），缺一不過。
5. **語氣**：讀起來像 `brand_tone` 描述的那個人在說話嗎？不像要指出是哪一句。
6. **構圖邏輯**：composition_prompt 有無視線／空間矛盾（人物所「看」的東西與其朝向一致嗎？
   是否塞了多個搶焦點的主行為）？對照 §3C-5 的自檢條。
7. **可信度支點**：每組至少一個具體支點（數字／社會證明／機制／保證）？只有空泛形容詞＝不過。
8. **利益翻譯**：賣點是否翻成「對客群的利益與意義」、而非規格陳述？
9. **鉤子前置／CTA**：`copy.primary_text` 第一行就勾人（前 ~125 字內、別被「⋯顯示更多」截掉）？CTA 強度對齊 `funnel_stage`（TOFU 軟／BOFU 硬）？
10. **設計準則 `design.md`（該品牌有提供才驗）**：composition_prompt 的配色／字體／版面／風格／避免項是否**遵循 design.md**？牴觸＝不過、要改到符合（design.md 只管視覺、**不蓋過合規**——合規仍以 #3 為最高）。

**整批層級**：是否至少 1–2 組**直接回應 §2 的「不買理由」**？沒有 → 指定改一組去回應抗拒點。

每組整組結論三選一：**通過 / 修正（列出改哪）/ 汰換**。

**不過關的回跳**（按失敗類型）：
- 合規不過 → 回 §3B/§3C 改寫該組文字與 prompt（必修）。
- 鉤子弱、語氣歪 → 回 §3B 重寫該組 copy/content（切角保留）。
- 多樣性撞車 → 回 §3B 以**未使用的（切角×hook×漏斗）組合**重生一組（維持 N 不變）。
- 可信度／利益翻譯／鉤子前置不足 → 回 §3B 補（加具體支點、把賣點翻成利益、鉤子提到第一行）。
- 圖面塞不下、構圖邏輯矛盾 → 回 §3C 修 content 槽位或 composition_prompt。
- design.md 不符 → 回 §3C 改 composition_prompt 對齊品牌設計準則。

**迴圈控制**：只把修過／重生的組再送 subagent 複驗，**最多 2 輪**；仍不過就標註問題
**交使用者裁決**（寧可少一組，不硬湊）。全部通過才進 §5 寫檔。
- **整批級修正要套用到每一組**：審稿若指出整批層級的問題（如槽位命名不一致、logo 版本/位置規範），
  **修正組與汰換重生的新組都必須先套用這些修正再送複驗**——否則同樣問題下一輪會被重複挑出，
  白燒複驗次數（實測踩過：汰換重寫的組漏掉整批的命名/版位約定，第二輪又被挑一次）。
回報使用者時附上驗證結論摘要。

### 5. 寫檔
結構與範例見 §3。依情況決定**追加**或**另開檔**：
- **新的一批**（全新 brief）：存成 `data/creatives/<id>.json`（`id` = 時間戳 + 品牌 slug，用 Bash 取）。
- **同一批再加**（「再多幾組 / 換切角」等同一 brief 的延伸）：**讀回該批 `<id>.json`，把新組 append 到 `creatives[]` 後面再寫回**，**不覆蓋、不另開檔**（原有的組保留）。
  - 新組**不要**自帶 `uid`/`images` 欄位（看板後端讀取時會自動補；自己編可能撞號）。
  - **並發契約（與 `generate-images` §4 一致，務必照做）**：`generate-images` skill 可能正在**另一個流程**把生成圖回寫進**同一份 `<id>.json`**（生圖在瀏覽器、回寫靠腳本，server 不知情、也擋不住）。追加時**絕不可拿稍早讀進記憶的快照整份覆寫**，否則會洗掉剛回寫的圖：
    - **append 的當下才重讀磁碟上最新的 `<id>.json`**，把新組接到 `creatives[]` 尾端後**原子寫入**（先寫 `.tmp` 再 `os.replace`）。
    - 既有組一律以磁碟為準、不要動（尤其別覆寫別組的 `images`/`uid`）。
    - 若使用者說「正在生圖」或你剛觸發過 `generate-images`，**先確認那批生圖已收尾再追加**，避免兩個流程同時寫同一檔。
- 換尺寸或不確定要不要併入同一批時，**先問使用者**。

### 6. 產後
回報：產了幾組、存到哪；模式 B 附**「本批核心命題」**＋（切角 × hook × 漏斗）一覽（並標各組的認知階段／回應的不買理由）、模式 A 附各文案的視覺版本數。
**自然的下一步——生圖**：告訴使用者可用 `/ad-generator:generate-images` 把這批產成主視覺（CDP 瀏覽器在 ChatGPT 生圖、回寫 JSON）。
也可主動問是否要：**再多幾組 / 換切角 / 換尺寸(default_aspect)**（「再多幾組 / 換切角」＝追加到同一批，見 §5）。
產出可在 `/ad-generator:serve` 看板檢視、編輯文案/構圖、刪除（**看板本身不生圖**，生圖在 generate-images）。
